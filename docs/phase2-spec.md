# フェーズ2 実装仕様

`phases.md`のフェーズ2(4項目)を、実装に着手できる粒度まで詳細化したもの。ドメインモデル・パッケージ構成・コーディング規約は`CLAUDE.md`に、コンセプト・AIのトーンは`mymanifesto.md`に準拠する。フェーズ1の実装(`phase1-spec.md`、実装済み)を土台とし、本書では**フェーズ1からの差分**のみを記述する。フェーズ1で確定した仕様(データモデルの基本構造・認証方針・つぶやきCRUD・当日日記生成の骨格)は、本書で明示的に変更すると書かない限りそのまま維持する。

対象範囲: つぶやきのカテゴリ自動分類、1日あたりの投稿上限、過去の日記一覧、日記の検索、遡り生成(任意日付での日記生成)。ユーザー登録UI・通知・カテゴリの手動修正・全文検索エンジン導入は対象外(8節参照)。

投稿上限は`phases.md`のフェーズ2項目には直接対応しないが、フェーズ1の「1日の投稿数は無制限」という方針(`phase1-spec.md`5節)を本書で上書きする形で導入する。異常な連続投稿(bot・誤操作等)を防ぐための実用上のガードレールであり、コンセプトの「SNS感覚で気軽に投稿する」という体験を損なわない範囲(50件/日)に設定する。

## 1. データモデルの変更点

### Post(`domain/Post.java`)への追加

| フィールド | 型 | 備考 |
|---|---|---|
| category | `PostCategory`(nullable) | AIによる自動分類結果。`@Enumerated(EnumType.STRING)`。未分類(分類前・分類失敗)は`null` |

`User`・`Diary`のテーブル定義はフェーズ1から変更しない。Diaryはフェーズ1と同様、Postへの直接の外部キーを持たない設計を維持する(カテゴリによる日記の絞り込みは今回スコープ外。8節参照)。

### PostCategory(`domain/PostCategory.java`、新規)

つぶやき1件につき必ず1カテゴリ(またはnull)。固定4種の`enum`とする。カテゴリを増減する場合はマイグレーションのCHECK制約と本enumを同時に変更する。

| 定数 | 表示ラベル | Bootstrapバッジクラス |
|---|---|---|
| `WORK` | 仕事 | `bg-primary` |
| `PRIVATE` | プライベート | `bg-success` |
| `LEARNING` | 学び | `bg-warning text-dark` |
| `OTHER` | その他 | `bg-light text-dark` |

表示ラベル・バッジクラスはenum定数のフィールドとして持たせ(コンストラクタ引数)、Thymeleaf側でenumのメソッド呼び出しにより取得する。

### Flywayマイグレーション(追加分)

- `V5__add_category_to_posts.sql` — `posts`テーブルに`category varchar(20)`(nullable)を追加する。CHECK制約は仕様確定前のドラフト値(8種)で作成された。
- `V6__narrow_posts_category_constraint.sql` — CHECK制約を確定仕様の4種(`CHECK (category IN ('WORK','PRIVATE','LEARNING','OTHER'))`)に絞り込む。V5は適用済みのため編集せず、新しいマイグレーションで訂正する(CLAUDE.mdの方針)。

新しいテーブルは作らない(過去の日記一覧・検索は既存の`diaries`テーブルへのクエリのみで実現するため。4節参照)。

## 4. 各機能の振る舞いと制約

### つぶやきのカテゴリ自動分類

- つぶやきの新規投稿・編集の**リクエスト処理をブロックしない**。投稿・編集のレスポンスを返した後、非同期(仮想スレッド)でAI分類を実行し、完了後に`category`列を更新する。「SNS感覚で気軽に投稿できる」というコンセプトの核(`mymanifesto.md`)を損なわないため、分類のためにつぶやき投稿自体を待たせることはしない。
- 実装は`PostService`内に`private final ExecutorService categorizationExecutor = Executors.newVirtualThreadPerTaskExecutor();`を保持し、`create`・`edit`の保存後にこのExecutorへ分類タスクをsubmitする形とする。CLAUDE.mdの方針通り、独自スレッドプールの管理を最小限に留めるため新規の`config`クラスは作らず`PostService`内に閉じる。アプリ終了時は`@PreDestroy`で`executor.shutdown()`を呼ぶ。
- 分類タスクは`PostCategorizer.categorize(String body)`を呼び出し、結果を`PostService.applyCategory(Long postId, PostCategory category)`(別トランザクション)で保存する。投稿がタスク実行前に削除されている場合は何もしない(`findById`が空なら早期return、例外にしない)。
- つぶやきを編集した場合は本文が変わるため再分類の対象とする。編集直後は`category`を一旦`null`に戻し(バッジを消す)、再分類の完了を待って新しいカテゴリを反映する。旧カテゴリのまま新本文を表示する期間を作らないための判断。
- 分類のAI呼び出しが失敗した場合(タイムアウト・レート制限等)は`category`を`null`のままにし、ログにwarnを出すのみでユーザー向けのエラー表示は行わない。バックグラウンド処理のためUI体験に影響を与えないことを優先し、自動リトライも行わない(日記生成と同じ「失敗時は次のアクションで再試行に委ねる」方針だが、こちらはユーザーが明示的に再試行する手段がないため、再編集されない限り未分類のままとなる。8節参照)。
- **カテゴリの手動変更**: AIの分類結果を、ユーザーが「今日のつぶやき」画面上のセレクトボックス(`PostCategory`の4値)から手動で変更できる。`POST /posts/{id}/category`(本人所有のみ、他人の投稿IDは404)で`category`を直接上書きする。手動変更は本文編集時の再分類まで維持される(本文を編集すると`category`が`null`にリセットされ、AIによる再分類が走るため、手動で選んだカテゴリは次の本文編集で上書きされうる。これは意図した挙動とする)。

### 投稿上限

- 1ユーザー・1日あたりの投稿数は50件を上限とする(`PostService`に`DAILY_POST_LIMIT`定数として持つ)。
- `PostService.create`で保存前に当日の投稿数をカウントし、上限に達していれば`DailyPostLimitExceededException`(`@ResponseStatus(HttpStatus.BAD_REQUEST)`)をスローして保存しない。
- `PostController.create`は本例外を捕捉し、`BindingResult.reject`でフォームのグローバルエラーとして扱う。バリデーションエラーと同じ「投稿フォームの直下にエラーメッセージを表示し、`posts/home`を再描画する」という体験に統一する(5節・6節のバリデーションエラー表示と同じ扱い)。
- 上限は新規投稿(`POST /posts`)のみに適用する。既存投稿の編集・削除は上限の対象外(件数を増やす操作ではないため)。
- 日付が変わればカウントは自動的にリセットされる(当日の投稿のみを対象にカウントするため、別途リセット処理は不要)。

### 過去の日記一覧・検索

- `GET /diaries`は常にログイン中ユーザー本人の日記のみを対象とする(`diaries.user_id`で絞り込み)。
- 一覧は`diaryOn`の降順(新しい日付が先頭)。1ページ20件、`page`クエリパラメータでページ送り(前へ/次へのリンクのみ。ページ番号ジャンプは持たない)。`DiaryRepository`に`findByUserIdOrderByDiaryOnDesc(Long userId, Pageable pageable)`を追加し`Page<Diary>`を返す。
- キーワード検索(`q`パラメータ)は日記本文(`body`)への部分一致・大文字小文字を区別しない検索とする。`DiaryRepository`に`findByUserIdAndBodyContainingIgnoreCaseOrderByDiaryOnDesc(Long userId, String q, Pageable pageable)`を追加する。全文検索エンジン(pg_trgm等)の導入は行わない(8節参照)。
- 各行に日付・本文の先頭80文字(超える場合は末尾に「…」)のプレビュー・`/diary/{date}`へのリンクを表示する。
- カテゴリによる日記の絞り込みは持たない。カテゴリ分類はつぶやき単位の機能であり、日記(Diaryは投稿へのFKを持たない設計)と直接結びつけない(8節参照)。

### 遡り生成(任意日付での日記生成)

- `POST /diaries/generate`に`date`(フォームパラメータ、任意)を追加する。省略時は当日として扱い、フェーズ1の挙動(`/posts`の生成ボタン)をそのまま維持する。
- `date`が未来の日付の場合は生成せず、フラッシュメッセージ「未来の日付の日記は作成できません」を出して`/diaries`へリダイレクトする。
- 指定日付のつぶやきが0件の場合は生成せず、「その日のつぶやきがありません」とフラッシュメッセージを出す。リダイレクト先は、`date`が当日ならフェーズ1と同じ`/posts`、当日以外なら`/diaries`とする(生成操作の起点画面に応じて戻り先を変える)。
- `PostService.findTodayPosts(Long userId)`は`findPostsOn(Long userId, LocalDate date)`に一般化し、当日分の取得は`findPostsOn(userId, LocalDate.now(clock))`の呼び出しに置き換える(呼び出し側の`DiaryController`を追従修正)。
- 生成・再生成に成功したら`/diary/{date}`(指定日の日記表示画面)へリダイレクトする。AI呼び出し失敗時の扱い(既存日記は変更しない、自動リトライしない、エラーメッセージを出す)はフェーズ1の方針をそのまま日付が当日以外の場合にも適用する。
- `GET /diary/{date}`は3つの状態を扱う。
  1. `date`が未来 → 404(`ResourceNotFoundException`)。
  2. その日の`Diary`が存在する → 本文を表示(フェーズ1と同じ)。
  3. `Diary`は存在しないが、その日の`Post`が1件以上存在する → 「この日の日記を作る」ボタン(`POST /diaries/generate`、`date`を隠しフィールドで送信)を表示する画面を返す(404にしない)。
  4. `Diary`も`Post`も存在しない → 404。
- 上記の判定のため`DiaryController.show`は`postService.findPostsOn(userId, date)`も呼び出す。3.の状態と1.(当日)の状態を区別する必要はなく、当日で日記未生成の場合も同じ「この日の日記を作る」表示で統一する(フェーズ1では`/posts`側にのみ生成導線があったが、フェーズ2では`/diary/{today}`に直接アクセスした場合も同様に生成導線を出せるようにする、という拡張であり後方互換を壊さない)。
- 遡り生成は既存のつぶやきを対象に日記を(再)生成するだけであり、過去日への新規つぶやき投稿・過去日のつぶやき編集/削除を可能にするものではない(2節参照)。

## 5. AI連携(`service/PostCategorizer.java`、新規)

- `service/AnthropicDiaryWriter.java`と同様、専用パッケージには分けず`service/`直下に置く。インターフェース`PostCategorizer`(`categorize(String body): PostCategory`)と実装`AnthropicPostCategorizer`に分離する構成も`DiaryWriter`/`AnthropicDiaryWriter`に揃える。
- モデル・トークン上限は日記生成と別の設定キーで持つ(分類は短い出力で済むため): `tsumory.ai.categorize-model`(デフォルト`claude-haiku-4-5`)、`tsumory.ai.categorize-max-tokens`(デフォルト64。tool useの出力はカテゴリ名自体は短いが、JSON構造のオーバーヘッドを見込んでこの値とする。小さすぎると`stop_reason=max_tokens`で`input`が生成される前に打ち切られ、分類に失敗する)。
- **ツール呼び出し(tool use)を強制する**: `category`という1つのenumプロパティ(4個の固定値)のみを入力スキーマに持つツールを1つ定義し、`tool_choice`でそのツールの呼び出しを強制する。自由記述テキストで返させて正規表現等でパースする方式は採らない(パース失敗によるカテゴリ欠落を防ぐため)。
- システムプロンプトは持たない、またはごく短いものに留める。ユーザーメッセージにはつぶやき本文のみを渡し、「以下のつぶやきに最も当てはまるカテゴリを1つ選んでください」のように指示する。日記生成のような「AIのトーン」(`mymanifesto.md`)を反映する必要はない(ユーザーに見える文章を生成する機能ではないため)。
- 判定に迷うような内容は`OTHER`(その他)を選ばせる。
- API呼び出し自体のログ方針(所要時間・トークン数のINFOログ、失敗時のWARNログ)は`AnthropicDiaryWriter`に準ずる。ただし本文はログに出さない(フェーズ1の「本文には個人的な内容が含まれるためログには文字数のみ出力する」方針を踏襲)。

## 6. 画面設計

### 今日のつぶやき画面(`posts/home.html`、変更)

- 各つぶやき行の本文の前(または後)に、`category`が設定されていればBootstrapバッジ(1節の表のクラス・ラベル)を表示する。`category`が`null`(未分類・分類中)の場合はバッジ自体を表示しない(「分類中」のような状態表示は行わない。次回のページ再読み込みで分類済みになっていれば自然に現れる)。

### 過去の日記一覧・検索画面(`diaries/index.html`、新規)

上から以下の順でセクションを配置する。

1. **検索フォーム**: キーワード入力欄(`q`)+検索ボタンのみ。`GET /diaries?q=...`へ送信(ページはリセットして0から)。
2. **日付を指定して日記を作る**: 日付入力(`<input type="date">`、上限を当日までに制限)+「作る/作り直す」ボタン。`POST /diaries/generate`に`date`を送信する。遡り生成(4節)の唯一のUI起点。
3. **一覧**: 日付降順。各行は日付・本文プレビュー(80文字)・「読む」リンク(`/diary/{date}`)。0件の場合(検索該当なし、または日記が1本もない)は空状態メッセージを表示する。
4. **ページネーション**: 「前へ」「次へ」のみ(`q`を維持したまま`page`を増減)。

### 日記表示画面(`diary/show.html`、変更)

- `Diary`が存在する場合はフェーズ1と同じ(本文・生成日時・「作り直す」ボタン)。「作り直す」も`date`を隠しフィールドで送信するよう変更する(当日固定ではなくなるため)。
- `Diary`が存在せずその日の`Post`が存在する場合は、本文表示の代わりに「この日はまだ日記がありません」+「この日の日記を作る」ボタン(`POST /diaries/generate`、`date`を隠しフィールドで送信)を表示する。
- いずれの状態でも「一覧へ戻る」リンク(`/diaries`)を追加する(ホーム`/posts`へのリンクはフェーズ1のまま維持)。

### 共通レイアウト(`fragments/layout.html`、変更)

- navbarに「過去の日記」リンク(`/diaries`)を追加する(ログイン中のみ表示)。

## 8. やらないこと(スコープ外)

- **カテゴリによる日記の絞り込み**: つぶやきのカテゴリ分類と日記検索を直接連携させない。Diaryは投稿へのFKを持たない設計(フェーズ1)を維持するための判断。将来必要になれば、生成時点のカテゴリ集合をDiary側にスナップショットするなどの設計を別途検討する。
- **全文検索エンジンの導入**: 日記検索は`LIKE`によるシンプルな部分一致に留める。データ量が増えて実用に耐えなくなった場合にpg_trgm等を検討する。
- **日記のカレンダー表示**: 月別カレンダーUIは持たず、シンプルな一覧+ページネーションのみ。
- **過去日への新規つぶやき投稿・過去日のつぶやき編集/削除**: つぶやきのCRUDは引き続き当日のみが対象(4節参照)。
- **カテゴリ分類失敗時の自動リトライ・ユーザーへの再試行導線**: つぶやきを編集すれば再分類が走るが、それ以外に明示的な再試行手段は用意しない。
- **通知・リマインド、ユーザー登録UI**: フェーズ1から引き続き対象外(`phases.md`参照)。
